# AshFeedback

> **Status: Proof of Concept.** APIs will change. Do not use in
> production. Validated inside a single host application (gs_net) —
> adoption elsewhere will surface edges.

Ash resource + triage state machine for bug-report feedback, built on
[`phoenix_replay`](https://github.com/jhlee111/phoenix_replay)'s ingest
pipeline.

## Demo

End-to-end flow — user toggles the theme, reports an issue through the
floating widget, and an admin replays the exact rrweb session from the
triage page:

[`docs/guides/demo-project.md`](docs/guides/demo-project.md).

## What you get

- A concrete `Feedback` resource exposed as a `use` macro — bring your
  own domain, repo, and `User` resource
- `AshStateMachine` triage workflow:
  ```
  new ─▶ acknowledged ─▶ in_progress ─▶ verified_on_preview ─▶ resolved
   │         │               │                    │
   └─────────┴───────────────┴────────────────────┴─▶ dismissed
  ```
- `AshPaperTrail` version history on every transition
- `Ash.Notifier.PubSub` broadcasts (`feedback:created`,
  `feedback:status_changed`, `feedback:assigned`, `feedback:verified`,
  `feedback:resolved`, `feedback:dismissed`, `feedback:comment_added`)
- Append-only `FeedbackComment` resource (no update, no destroy — edits
  happen via new comments)
- `AshPrefixedId` support for the assignee / author FK columns
- Code interface ready for AshAi, JSON:API, GraphQL, and the
  `gsnet-deploy`-style deploy-pipeline auto-resolve flow

## Requirements

- [`phoenix_replay`](https://github.com/jhlee111/phoenix_replay)
  installed and running (its `mix phoenix_replay.install` migration
  creates the `phoenix_replay_feedbacks` /
  `phoenix_replay_feedback_comments` tables that this library binds
  to)
- Elixir 1.14+
- Ash 3.x + AshPostgres + AshStateMachine + AshPaperTrail

## Installation

### 1. Install phoenix_replay first

Follow [`phoenix_replay`'s README](https://github.com/jhlee111/phoenix_replay#installation)
through `mix phoenix_replay.install` + `mix ecto.migrate`. Stop before
you set the `:storage` config key — this library provides it.

### 2. Add the dep

```elixir
# mix.exs
def deps do
  [
    {:phoenix_replay, github: "jhlee111/phoenix_replay", branch: "main"},
    {:ash_feedback, github: "jhlee111/ash_feedback", branch: "main"}
  ]
end
```

### 3. Point `:phoenix_replay` at the Ash storage module

```elixir
# config/config.exs (or per-env)
config :phoenix_replay,
  # ... identify, metadata, session_token_secret as usual ...
  storage: {AshFeedback.Storage, resource: MyApp.Feedback.Entry, repo: MyApp.Repo}
```

`AshFeedback.Storage` is a thin implementation of the
`PhoenixReplay.Storage` behaviour — feedback writes go through your
Ash domain (policies, PaperTrail, prefixed IDs all apply), rrweb
events still stream to raw Ecto for throughput.

### 4. Create your Feedback domain

```elixir
# lib/my_app/feedback.ex
defmodule MyApp.Feedback do
  use Ash.Domain, otp_app: :my_app

  resources do
    resource MyApp.Feedback.Entry
    resource MyApp.Feedback.Entry.Version  # auto-generated by AshPaperTrail
    resource MyApp.Feedback.Comment        # optional — skip if you don't need comments
  end
end
```

### 5. Create your concrete Feedback resource

```elixir
# lib/my_app/feedback/entry.ex
defmodule MyApp.Feedback.Entry do
  use AshFeedback.Resources.Feedback,
    domain: MyApp.Feedback,
    repo: MyApp.Repo,
    assignee_resource: MyApp.Accounts.User,
    pubsub: MyApp.PubSub
end
```

That's it — 5 lines of `use` opts give you a full Ash resource with
the full triage state machine.

**Host options:**

| Option | Required | Purpose |
|--------|----------|---------|
| `:domain` | yes | Your Ash.Domain module |
| `:repo` | yes | Ecto repo |
| `:assignee_resource` | no | Your User module. If omitted, the `assignee` / `verified_by` / `resolved_by` relationships are skipped. |
| `:assignee_attribute_type` | no* | FK column type — see gotcha below |
| `:pubsub` | no | `Phoenix.PubSub` name. Omit to disable notifications. |
| `:paper_trail_actor` | no | Enable actor tracking on versions — see gotcha below |
| `:prefix` | no | AshPrefixedId prefix (default `"fbk"`) |

### 6. (Optional) Create the Comment resource

```elixir
# lib/my_app/feedback/comment.ex
defmodule MyApp.Feedback.Comment do
  use AshFeedback.Resources.FeedbackComment,
    domain: MyApp.Feedback,
    repo: MyApp.Repo,
    feedback_resource: MyApp.Feedback.Entry,
    author_resource: MyApp.Accounts.User,
    pubsub: MyApp.PubSub
end
```

### 7. Run codegen + migrate

The library's base schema is owned by `phoenix_replay`'s installer, so
you don't need a new migration for the tables themselves. But
AshPaperTrail generates its own `_versions` table — run codegen to
emit that:

```bash
mix ash.codegen add_feedback_paper_trail
mix ash.migrate
```

## Code interface

After the `use` macro runs, your concrete Feedback resource exposes:

```elixir
# Submit (called by PhoenixReplay.Storage when the widget POSTs /submit)
MyApp.Feedback.Entry.submit!(%{session_id: ..., description: ...})

# Triage transitions
MyApp.Feedback.Entry.acknowledge!(feedback_id, actor: user)
MyApp.Feedback.Entry.assign!(feedback_id, %{assignee_id: user_id}, actor: user)
MyApp.Feedback.Entry.verify!(feedback_id, %{pr_urls: [...]}, actor: user)
MyApp.Feedback.Entry.resolve!(feedback_id, %{}, actor: user)
MyApp.Feedback.Entry.dismiss!(feedback_id, %{reason: :not_a_bug}, actor: user)

# Queries
MyApp.Feedback.Entry.list_feedback!(severity: :high, limit: 20)
MyApp.Feedback.Entry.get_feedback!(id)

# Deploy-pipeline hooks (e.g. preview→prod promote auto-resolve)
MyApp.Feedback.Entry.list_preview_blockers!()      # open rows reported on preview
MyApp.Feedback.Entry.list_verified_non_preview!()  # rows waiting on ship
MyApp.Feedback.Entry.promote_verified_to_resolved!(
  %{promoted_at: DateTime.utc_now()},
  actor: deploy_pipeline_user
)
```

## PubSub topics

When `:pubsub` is set, notifications publish to these topics (one
broadcast each):

| Topic | Fires on |
|-------|----------|
| `feedback:created` | `:submit` |
| `feedback:status_changed` | every transition |
| `feedback:assigned` | `:assign` |
| `feedback:verified` | `:verify` |
| `feedback:resolve` | `:resolve` |
| `feedback:dismissed` | `:dismiss` |
| `feedback:comment_added` | new `FeedbackComment` |

> **Gotcha**: Ash's PubSub notifier joins a multi-segment list into
> one topic. `publish :assign, ["status_changed", "assigned"]` fires
> **only** `feedback:status_changed:assigned`, not both of the above.
> To fan out, the library duplicates `publish` calls. If you layer
> your own, do the same.

## Gotchas (worth reading before integrating)

### 1. `AshPrefixedId` hosts must pass concrete FK type

If your User resource uses `AshPrefixedId`, pass the concrete ObjectId
type for the assignee FK:

```elixir
use AshFeedback.Resources.Feedback,
  # ...
  assignee_resource: MyApp.Accounts.User,
  assignee_attribute_type: MyApp.Accounts.User.ObjectId  # ← required
```

The default short-name `:uuid` resolves to
`AshPrefixedId.AnyPrefixedId` on such hosts, which round-trips a
prefixed string into a raw UUID and breaks the `belongs_to` load.
Same rule applies to `:author_attribute_type` on `FeedbackComment`.

### 2. PaperTrail FK serialization

`AshPaperTrail` serializes diffs as JSONB. `AshPrefixedId` FK columns
(`:assignee_id`, `:verified_by_id`, `:resolved_by_id`, `:related_to_id`)
don't have a JSON-safe `dump_to_embedded` path, so the library lists
them in `paper_trail do ignore_attributes [...] end`. State
transitions still capture who-did-what via `:status` + action name —
nothing meaningful is lost.

### 3. Actor tracking needs one more opt + a migration

Out of the box, versions record the action name and attribute diff
but not the acting user. Enable actor tracking by passing
`paper_trail_actor: MyApp.Accounts.User` to the macro AND running
`mix ash.codegen` afterward — AshPaperTrail adds a `user_id` column
to the versions table.

## Audio narration (optional)

ADR-0001 — voice narration on feedback submissions. The reporter taps
🎙 in the widget panel, records a short clip, and submits; the audio
file rides through [AshStorage](https://github.com/ash-project/ash_storage)
(presigned upload to S3 or any compatible backend) and links to the
feedback row. The admin replay view (Phase 3, in progress) plays the
clip in sync with the rrweb cursor.

**Status**: Phases 1 + 2 shipped — host opt-in, recorder UI, prepare
endpoint, submit-side wiring. Phase 3 (admin playback synced to
phoenix_replay's timeline event bus, ADR-0005) is the next deliverable.

> **Mode availability**: the audio recorder is registered with
> `modes: ["on_demand"]`. It mounts only on widgets whose `recording` is
> `:on_demand` ("Record-and-report mode"). On `:continuous` widgets the addon
> is silently skipped — voice commentary on cached/retrospective replays cannot
> be synced to the rrweb timeline. Control style (`:float` / `:headless`) is
> independent of this filter. See phoenix_replay's mode-aware panel-addons
> spec (`docs/superpowers/specs/2026-04-25-mode-aware-panel-addons.md`) for
> the full IA framework.

### Enabling

```elixir
# config/config.exs
config :ash_feedback,
  audio_enabled: true,
  feedback_resource: MyApp.Feedback.Entry,
  audio_attachment_resource: MyApp.Storage.Attachment,
  audio_max_seconds: 300  # default
```

### Host requirements

- `{:ash_storage, github: "ash-project/ash_storage"}` in your deps
  (pre-Hex; switch to `~> 0.1` once it cuts a Hex release).
- Host-defined `Blob` + `Attachment` AshStorage resources. See
  [`dev/resources/blob.ex`](https://github.com/ash-project/ash_storage/blob/main/dev/resources/blob.ex)
  and
  [`dev/resources/attachment.ex`](https://github.com/ash-project/ash_storage/blob/main/dev/resources/attachment.ex)
  in the AshStorage repo for reference shapes (Phase 5f's installer
  will scaffold these).
- An AshStorage service configured for the Blob resource — `Disk`
  for dev, `S3` (or compatible) for prod.

### Pass the audio resources to the macro

```elixir
defmodule MyApp.Feedback.Entry do
  use AshFeedback.Resources.Feedback,
    otp_app: :my_app,
    domain: MyApp.Feedback,
    repo: MyApp.Repo,
    audio_blob_resource: MyApp.Storage.Blob,
    audio_attachment_resource: MyApp.Storage.Attachment
end
```

`otp_app:` is required when audio is enabled so AshStorage's per-resource
service config (`config :my_app, MyApp.Feedback.Entry, storage: [...]`)
resolves at runtime. Audio-disabled hosts can leave it off.

Run `mix ash.codegen audio_storage` to generate the migrations for the
two storage tables.

### Wiring the browser side

In your root layout — load the recorder CSS in `<head>` and the
recorder JS at the **end** of `<body>`, after the `phoenix_replay`
widget element so the addon registers before `phoenix_replay`'s panel
mounts:

```heex
<link rel="stylesheet" href={~p"/assets/ash_feedback/audio_recorder.css"} />
...
<PhoenixReplay.UI.Components.phoenix_replay_widget ... />
<script defer src={~p"/assets/ash_feedback/audio_recorder.js"}></script>
```

Add a `Plug.Static` entry serving `ash_feedback/priv/static/assets`:

```elixir
plug Plug.Static,
  at: "/assets/ash_feedback",
  from: {:ash_feedback, "priv/static/assets"}
```

In your router, mount the prepare endpoint inside an authenticated
browser scope:

```elixir
import AshFeedback.Router, only: [audio_routes: 0]

scope "/" do
  pipe_through :browser
  audio_routes()  # or audio_routes(path: "/api/audio")
end
```

### Browser support

- Chrome / Firefox / Edge — `audio/webm; codecs=opus` (primary).
- Safari — `audio/mp4; codecs=mp4a.40.2` (fallback).
- No supported codec → mic button disabled with a tooltip; the rest
  of the form remains usable.
- Microphone permission denied → inline notice; the user can still
  submit without audio.

Audio behavior is opt-in at every layer — hosts who skip the config
flag get the existing description-only feedback flow with zero
change.

## Deploy pipeline integration

The two `list_*` read actions + `promote_verified_to_resolved` generic
action are the hooks for a CI/CD tool to query before a deploy and
auto-resolve verified rows after a successful ship. In gs_net these
are wired to three HTTP endpoints under `/api/internal/feedback/*`
with bearer-token auth — see the gs_net Phase 5d plan
(`docs/plans/active/2026-04-21-feedback-triage.md`) for a reference
implementation.
